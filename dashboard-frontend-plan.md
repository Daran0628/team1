# 프론트엔드 구성 계획 (대시보드 & 공통 레이아웃)

`sitemap.png`(전체 메뉴 구조)와 `wireflame.png`(대시보드 와이어프레임)를 기준으로 정리한
프론트 기본 틀 구축 계획.

## 1. 배경

- 현재 로그인 이후 진입점인 `index.html`이 비어있고, 각 페이지(`group.html`, `mail.html`,
  `rbac.html` 등)는 공통 네비게이션 없이 페이지별 브레드크럼 헤더만 가지고 있음.
- `sitemap.png`에 정의된 전체 메뉴 구조를 기준으로, 로그인 후 모든 페이지에서 공유할
  **좌측 사이드바 + 상단바 셸**과, 그 진입점이 될 **대시보드 페이지**를 새로 만든다.

## 2. 사이트맵 대비 현재 구현 현황

| 사이트맵 섹션 | 하위 항목(사이트맵 기준) | 구현 상태 | 관련 라우트/파일 |
|---|---|---|---|
| 로그인 | - | ✅ | `/login`, `login.html` |
| 대시보드 | - | ❌ (이번 작업 대상) | `index.html` (빈 파일) |
| 헬프데스크(티켓) | 티켓 목록/상세/등록/내 티켓/통계 | ❌ 미구현 | - |
| FAQ | FAQ 목록/상세/검색 | ❌ 미구현 | - |
| 공지사항 | 공지사항 목록/상세 | ❌ 미구현 | - |
| 메일 서비스 | 받은/보낸함, 작성, 상세, 검색 | ✅ | `/mail`, `MailRestController.py` |
| Chatting | 채팅 목록/방/메시지/파일 공유 | ✅ | `/chat`, `/chat/<room_id>`, `ChatRestController.py` |
| 공유 스토리지 | 파일 목록/업로드/다운로드/공유/휴지통 | ✅ | `/objstorage`, `StorageRestController.py` |
| VDI | 목록/상세/시작·종료/스냅샷 | ✅ | `/vdi`, `/vdi/list`, `VdiRestController.py` |
| 코딩테스트 | 목록/상세/응시/결과 | ❌ 미구현 | - |
| 인물 검색 | 인물 검색/상세/부서별 조회 | 🟡 부분 구현 (검색만) | `/person`, `MemberRestController.py` |
| 마이페이지 | 정보 조회/수정/비밀번호/로그인 내역 | 🟡 부분 구현 (조회/수정만) | `/mypage`, `MemberRestController.py` |
| 관리자 페이지 | 사용자/부서/권한/공지/FAQ/티켓/설정/로그 관리 | 🟡 부분 구현 (권한 관리만) | `/iam`, `/iam/roles`, `/iam/groups`, `/iam/permissions` |

**정리**: 대시보드, 헬프데스크, FAQ, 공지사항, 코딩테스트는 완전히 새로 만들어야 하고,
인물검색/마이페이지/관리자 페이지는 사이트맵 기준으로 보면 하위 기능이 더 남아있음.
이번 작업 범위는 우선 **공통 레이아웃 셸 + 대시보드 정적 틀**로 한정한다.

## 3. 대시보드 와이어프레임 분석 (`wireflame.png`)

- **좌측 사이드바**: 로고 + 프로젝트명, 사이트맵의 11개 메뉴(대시보드/헬프데스크/FAQ/공지사항/
  메일/Chatting/공유 스토리지/VDI/코딩테스트/인물 검색/마이페이지/관리자 페이지), 하단 로그아웃
- **상단바**: 우측에 사용자 이름 + 드롭다운
- **메인 컨텐츠 (3단 구성)**
  1. 환영 카드(프로필, "OO님 환영합니다", 소속/직급/이메일/마지막 로그인) + 바로가기 카드(8개 아이콘 그리드)
  2. 3열 그리드: 나의 티켓 현황 / 공지사항 / 메일 요약
  3. 3열 그리드: 최근 Chatting / FAQ / 게시판

## 4. 공통 레이아웃 구조 설계

정적 HTML(Flask + vanilla JS) 구조를 유지하는 한도 내에서, 페이지마다 사이드바 마크업을
복붙하지 않도록 공통 셸을 분리한다.

- `static/partials/sidebar.html` — 사이드바 마크업 (메뉴 항목 + active 상태 표시용 data 속성)
- `static/js/layout.js` — 공통 로직
  - `fetch('/static/partials/sidebar.html')` 후 `#sidebar-slot`에 주입
  - 현재 경로(`location.pathname`) 기준으로 활성 메뉴 하이라이트
  - 로그아웃 버튼 → 기존 `/api/auth/logout` 연동
  - 상단바 사용자 이름 → JWT payload(`sub`)에서 표시 (mail.js의 `parseJwt` 패턴 재사용)
- `static/css/layout.css` — 사이드바/상단바 레이아웃 스타일
- 각 페이지는 `<div id="sidebar-slot"></div>` + `<script src="/static/js/layout.js"></script>`만
  추가하면 공통 셸을 갖추도록 함 (기존 `group.html` 등 페이지에도 점진 적용)

> CSP가 `script-src 'self'`라 인라인 스크립트/스타일 불가 — 반드시 별도 `.js`/`.css` 파일로 분리.

## 5. 대시보드 카드별 데이터 소스

| 카드 | 데이터 소스 | 비고 |
|---|---|---|
| 환영 카드 (소속/직급/이메일/마지막 로그인) | `GET /api/member/me` | 이미 구현됨, 그대로 재사용 |
| 메일 요약 (받은/보낸/임시보관/중요) | `GET /api/mail/inbox`, `/sent` | "임시보관함/중요 메일" 개념은 백엔드에 없음 → 범위 축소 필요 |
| 최근 Chatting | Chat 목록 API | 기존 `ChatRestController` 활용 |
| 나의 티켓 현황 | - | 백엔드 없음 → 신규 기능, 우선 빈 상태(placeholder)로 표시 |
| 공지사항 | - | 백엔드 없음 → 신규 기능, placeholder |
| FAQ | - | 백엔드 없음 → 신규 기능, placeholder |
| 게시판 | - | 사이트맵에 없는 항목 (와이어프레임에만 존재) → 범위/의미 확인 필요 |
| 바로가기 아이콘 8개 | 정적 링크 | 각 섹션 라우트로 연결 |

## 6. 진행 순서 제안

1. **공통 레이아웃 셸** 구현 (`sidebar.html` + `layout.js` + `layout.css`), 대시보드 페이지에 우선 적용
2. **대시보드 정적 틀** 구현 — 실데이터 없이 wireflame 레이아웃 그대로 (카드 자리만 확보, placeholder 텍스트)
3. **이미 구현된 API 연동** — 환영 카드(`/api/member/me`), 메일 요약, 최근 Chatting
4. **미구현 섹션은 빈 상태 처리** — 티켓/공지사항/FAQ/게시판 카드는 "준비 중" 표시로 남기고 별도 작업으로 분리
5. **기존 페이지에 공통 셸 점진 적용** — group/mail/chat/objstorage/vdi/person/mypage/iam 순서로 사이드바 붙이기 (대시보드 검증 후 진행)

## 7. 확인 필요 사항

- "게시판" 카드가 사이트맵에 없는 항목인데 실제로 만들 기능인지, 공지사항과 통합할지
- 사이드바 메뉴명과 실제 라우트 매핑 (예: "관리자 페이지" 클릭 시 `/iam`으로 연결할지, 새 관리자 허브를 만들지)
- 헬프데스크(티켓)/FAQ/공지사항/코딩테스트 4개 신규 기능의 우선순위와 일정
